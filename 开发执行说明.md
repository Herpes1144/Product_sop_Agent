# 开发执行说明

## 1. 开发目标
将当前静态原型升级为“mock 数据驱动 + 真实 AI 分析 + 动作执行 + 状态流转 + 二次分析”的闭环版 MVP。

## 2. 开发优先级
1. 工单切换 + Agent 区真实结构化渲染
2. 动作按钮生效 + 状态/记录联动
3. 补材料后二次分析
4. 聊天区和演示细节优化

## 3. 首批联调 Case
优先联调：
1. CASE-01：补图后二次推进
2. CASE-02：信息充分直接进入继续处理路径
3. CASE-03：高风险升级路径
4. CASE-08：描述不符争议路径

不建议作为首批联调：
- CASE-06
- CASE-09
- CASE-10

## 4. 核心 Case 摘要
### CASE-01
- 问题：杯身破裂，无图
- 预期：
  - 第一轮：`request_photo` → `waiting_material`
  - 第二轮：补图后 `continue_refund` → `continue_path`

### CASE-02
- 问题：平板碎屏，图片齐全
- 预期：
  - 直接 `continue_return_refund` → `continue_path`

### CASE-03
- 问题：暖风机破损且疑似漏电
- 预期：
  - `escalate` → `waiting_escalation`

### CASE-08
- 问题：家具尺寸与宣传不一致，截图不完整
- 预期：
  - 第一轮：`request_screenshot` → `waiting_material`
  - 第二轮：补截图后 `continue_return_refund` → `continue_path`

## 5. 动作优先级规则
1. `should_escalate = true` 时，`escalate` 优先于 continue 类动作
2. `need_more_materials = true` 且问题未明确时，补材料类动作优先
3. 信息充分且无需升级时，continue 类动作作为主推荐动作
4. `mark_resolved` 仅在人工明确可结案时使用
5. `reply_suggestion` 仅为沟通辅助能力，不参与主动作竞争

## 6. 动作执行要求
### 6.1 request_video / request_photo / request_screenshot
执行后必须：
- 更新结果类型为 `waiting_material`
- 新增处理记录
- 自动填充聊天建议
- 等待人工点击“重新分析”

### 6.2 escalate
执行后必须：
- 更新结果类型为 `waiting_escalation`
- 新增处理记录
- 自动填充升级说明
- 工单列表高亮风险/优先级

### 6.3 continue 类动作
执行后必须：
- 结果类型进入 `continue_path`
- 展示当前推荐路径标签（退款/退货退款/换货/补发）
- 新增处理记录
- 自动填充沟通建议

### 6.4 mark_resolved
执行后必须：
- 更新状态为 `resolved`
- 新增结案记录
- 分析区进入只读或结束态

## 7. 工单列表与页面联动
### 工单切换
- 先刷新原始信息区
- 再刷新 Agent 区
- 再同步聊天区与处理记录区

### 动作点击
至少触发以下三项中的三项：
- 状态/结果变化
- 聊天区变化
- 处理记录变化
- AI 分析区变化
- 工单列表摘要变化

## 8. 重新分析规则
- 统一由人工点击“重新分析”触发
- 客户上传材料本身不自动触发
- 二次分析要带上新附件、最新聊天、最新处理记录、当前工单状态

## 9. 处理记录文案规则
- AI 判断类：`AI 建议…… / AI 判断……`
- 人工执行类：`售后已执行…… / 售后已发送……`
- 状态变化类：`工单进入…… / 当前路径进入……`
- 一条记录只表达一件事

## 10. 前端展示规则
- 主推荐动作：主按钮
- 辅助动作：次级按钮
- `reply_suggestion`：独立话术辅助按钮
- `continue_path` 不作为页面状态展示，而显示为“退款路径中 / 换货路径中”等

## 11. 当前版本约束
- 不接真实企业系统
- 不做自动审批
- 不做自动退款/自动退货/自动换货
- 不做完整视频理解作为首版阻塞项

## 12. Codex 实现建议
### 第一步
先实现 `/api/ai/analyze-ticket`，返回标准结构化 JSON。

### 第二步
把 Agent 区从 mock 切换为真实结构化返回。

### 第三步
把动作按钮接上：
- 状态/结果更新
- 处理记录新增
- 聊天建议填充
- 工单列表摘要刷新

### 第四步
实现补材料后的“重新分析”。

### 第五步
最后补 toast、loading、错误兜底和演示细节。
