## 二、产品介绍

### 2.1 产品定位

本产品是一个面向企业内部一线售后处理人员的商品质量投诉分流 Agent 工作台，用于帮助售后人员在处理模糊、证据不完整、规则边界复杂的质量投诉时，更快完成问题理解、问题分类、SOP 匹配、处理建议获取与下一步动作判断。

第一版产品不做完整售后工单平台，不做真实系统接入，不做自动审批和自动执行闭环，仅围绕“商品质量投诉分流”场景，验证“识别—匹配—建议—动作推荐”的最小闭环。

### 2.2 产品综述

当前版本 PRD 以 mock 前端原型为页面真值，用于对齐已经实现的工作台结构、模块内容与交互方式。原型采用左侧工单列表、中间原始信息与聊天工作区、右侧 Agent 辅助区的三栏工作台结构，使用 mock 工单、订单、商品、聊天、处理记录与建议动作完成交互验证。

### 2.3 需求分析

#### 业务背景

在企业售后场景中，商品质量投诉是一个高频但判断复杂的问题类型。一线售后人员在接到客户投诉后，通常需要先阅读客户原始表述，再核对订单与商品信息，检查图片、视频等证据材料，随后结合售后规则与 SOP 判断该问题是否属于质量问题、是否仍在处理时效内、是否需要补材料，以及是否需要升级处理。当前流程并不是单纯的机械执行，而是高度依赖经验和规则理解。

对于成熟售后人员来说，这类问题可以依靠经验快速判断；但对于新人或经验不足的一线处理人员而言，常见困难包括：客户投诉表述模糊、证据不完整、规则边界不清、不同问题类型容易混淆，以及不确定当前情况是否应该继续处理还是升级处理。这会导致处理效率下降，也容易产生判断不一致、漏步骤、漏记录和升级不及时等问题。

因此，本项目第一版不试图重做完整售后工单系统，也不追求自动替代人工决策，而是选择“商品质量投诉分流”作为切入场景，构建一个面向一线售后人员的 AI 辅助工作台，优先帮助其完成投诉理解、问题分类、SOP 匹配、补材料提示、处理建议输出和下一步动作推荐，从而验证 AI 在企业售后流程中的辅助分诊价值。

#### 用户侧痛点

- 新人遇到质量投诉时，需要频繁翻规则手册或询问老员工
- 客户投诉内容常常模糊，难以快速识别问题类型
- 证据不全时，售后不知道该补哪些材料
- 规则边界复杂时，售后不确定是否应升级处理
- 同类投诉在不同人员手中容易出现判断不一致和漏步骤

#### 产品机会

- 将模糊投诉内容结构化
- 将规则与 SOP 的判断依据显性化
- 将“补材料 / 升级 / 继续处理”做成标准化路径建议
- 降低一线售后处理质量投诉时的判断成本和沟通成本

## 四、全局说明

### 4.1 页面范围说明

第一版页面聚焦一个核心工作台页面，即“质量投诉分流工作台”。

页面围绕工单进入后的信息查看、AI 分析、建议动作和售后回复展开，不扩展到完整后台管理系统。

### 4.2 全局状态说明

当前版本统一使用以下处理状态：

- 待处理
- 待补材料
- 待升级
- 已标记处理

详细字段与状态定义见《03A_结构化字段与状态表》。

### 4.3 常用交互说明

- 用户通过左侧工单列表浏览和搜索工单
- 点击工单后，中间原始信息区优先刷新，再触发右侧 Agent 区分析刷新
- 右侧 Agent 区以“AI问题摘要 / SOP判断和推荐 / 快捷操作”三段式展示建议
- 快捷操作可将推荐话术写入聊天输入框，也可触发待补材料、待升级、已标记处理等状态更新
- 中间历史记录卡片默认收起，展开后可通过 tab 切换查看“历史聊天记录 / 处理记录”
- 原始投诉卡片右侧展示附件缩略图，点击可打开附件预览弹层
- 聊天窗口作为独立模块展示，支持输入、发送，以及通过分隔条拖拽调整区域高度
- 发送消息后，聊天记录、处理记录和 Agent mock 分析会同步刷新

## 七、功能清单

### 7.1 工单展示

- 模块目标：承载当前待处理质量投诉工单的入口
- 核心内容：工单列表、单号、时间、优先级、问题类型、处理状态
- 支持功能：搜索工单、点击切换、空结果提示
- 用户价值：帮助售后快速定位当前要处理的投诉

### 7.2 商品原始信息展示

- 模块目标：展示与投诉相关的原始事实信息
- 核心内容：客户原始投诉内容、附件展示区、商品信息、订单信息、处理状态
- 用户价值：辅助判断问题类型与处理路径，避免只依赖 AI 结论

### 7.3 历史记录区

- 模块目标：承载当前工单的历史上下文
- 核心内容：历史聊天记录、处理记录
- 支持功能：默认收起、展开后通过 tab 切换查看
- 用户价值：在不打扰首屏主信息的前提下补充上下文

### 7.4 Agent 问题识别结果

- 模块目标：将客户原始投诉结构化
- 核心内容：AI问题摘要、问题分类、当前处理状态提示
- 用户价值：降低一线售后理解问题的成本

### 7.5 SOP 判断依据与建议动作

- 模块目标：让一线人员看到“为什么这样建议”
- 核心内容：SOP判断依据、判断结论、建议动作说明列表
- 用户价值：提升处理一致性与可解释性

### 7.6 快捷操作推荐

- 模块目标：将常见动作标准化
- 核心内容：推荐回复、补视频、补照片、补截图、升级处理、继续处理路径、标记已处理
- 交互方式：点击后可将推荐话术填入输入框，或触发状态更新
- 用户价值：减少重复操作，提高处理效率

### 7.7 客户沟通区

- 模块目标：承载当前轮次的客户沟通
- 核心内容：当前会话、输入框、发送动作、状态提示
- 支持功能：输入和发送消息、结合 AI 推荐回复编辑发送、拖拽调整聊天区高度
- 用户价值：让售后可在同一页面内完成判断到沟通的闭环

## 八、页面功能详细说明

### 8.1 质量投诉分流工作台（主页面）

#### 页面目标

帮助一线售后在一个工作台页面内完成：

- 查看投诉原始信息
- 获取 AI 识别与判断结果
- 执行下一步动作
- 回复客户并更新处理状态

#### 页面主入口

首页以工单列表为主入口。用户先从左侧工单列表中选择需要处理的质量投诉工单，再进入具体处理视图。切换工单后，先加载中间原始信息区，再加载 Agent 辅助区；客户沟通区随当前工单上下文同步显示。

#### 页面布局结构

页面整体采用四区结构：

- 左侧：工单展示侧边栏
- 中间上半区：工单原始信息区
- 右侧：Agent 辅助区
- 中间下半区：客户沟通区

#### 页面首屏 P0 信息

- 客户原始投诉内容
- 商品信息
- 订单信息 / 处理状态
- AI 问题识别结果
- SOP 判断依据 / 建议动作
- 客户沟通区

#### 次级 P1 信息

- 历史聊天记录
- 处理记录
- 附件预览

### 8.2 工单展示侧边栏

#### 页面目标

作为工作台首页的主入口，承载待处理投诉工单的浏览与切换。

#### 展示字段

- 单号
- 处理状态
- 问题类型
- 时间
- 优先级

#### 支持功能

- 搜索工单
- 浏览工单卡片
- 点击切换当前工单
- 搜索无结果时展示空态提示

#### 设计原则

- 工单以卡片展示，占用空间尽量收窄，以展现更多工单
- 点击后切换当前工单视图，并刷新页面其他区域内容

#### 用户价值

帮助售后快速从多条投诉中进入具体处理对象。

### 8.3 中间原始信息区

#### 页面目标

集中展示当前工单的原始输入信息，保证售后可以先看“事实”，再看 AI 判断。

#### 区域内容

##### 1）客户原始投诉内容与附件展示区

- 展示客户本次投诉的原始文本
- 作为页面首屏优先信息
- 卡片右侧展示附件缩略图
- 附件存在时可点击预览，空位显示“待补充”

##### 2）商品信息

- 商品名称
- 型号 / 规格
- 类目
- 收货时间
- 是否高风险商品

##### 3）订单信息 / 处理状态

- 订单号
- 订单状态
- 当前处理状态（待处理 / 待补材料 / 待升级 / 已标记处理）

##### 4）历史记录与处理记录

- 以共用卡片形式呈现
- 默认态为收起态，点击“展开”后显示详细内容
- 展开后通过 tab 切换“历史聊天记录 / 处理记录”
- 作为次级信息，不抢首屏主信息

#### 设计补充

- 原始信息区以多卡片结构呈现：投诉与附件、商品信息、订单信息、历史记录
- 点击工单后，中间原始信息区先进入加载态，加载提示为“加载中”

#### 用户价值

让售后可以基于原始事实理解当前投诉，避免只依赖 AI 结论。

### 8.4 Agent 辅助区

#### 页面目标

输出 AI 对当前投诉的结构化理解、判断依据与下一步建议。

#### 区域内容

##### 1）AI问题摘要

- 当前问题分类
- 当前处理状态
- AI问题摘要

##### 2）SOP判断和推荐

- 当前命中的规则
- 适用的 SOP 逻辑
- 关键判断说明
- 当前判断结论（如建议继续处理 / 建议先补充材料 / 建议升级处理）
- 建议动作说明列表

##### 3）快捷操作

- 推荐回复
- 补视频
- 补照片
- 补截图
- 升级处理
- 继续处理路径
- 标记已处理

#### 设计原则

- Agent 区不仅要告诉用户“做什么”，还要告诉用户“为什么这样建议”
- 在原始信息区完成加载后，Agent 再进入分析态，提示为“分析中”
- 切换工单和发送客户回复后，Agent 区均会基于最新上下文进行 mock 分析刷新
- AI 建议是“推荐”，不是“自动执行”

#### 用户价值

降低判断成本，让一线售后看到 AI 的可解释建议，而不是一个黑盒结论。

### 8.5 客户沟通区

#### 页面目标

承载当前工单下的实时客户沟通。

#### 与历史聊天记录的区别

- 历史聊天记录：用于查看上下文和过往沟通
- 客户沟通区：用于当前轮次回复与操作

#### 支持功能

- 输入和发送消息
- 结合 AI 推荐回复进行处理
- 快捷操作将推荐话术写入输入框，再由人工编辑发送
- 通过分隔条拖拽调整聊天区高度

#### 当前模块形态

- 聊天区为独立模块
- 顶部带有“客户沟通区”标题条
- 标题条下方提供拖拽分隔条
- 不提供展开 / 收起 / 默认高度按钮组

#### 用户价值

让售后在不跳出页面的情况下完成判断到沟通的闭环。

- 问题分类

  - 当前原型默认不展示

详细字段定义见《03A_结构化字段与状态表》。

### 9.3 AI核心判断逻辑

AI 在本场景中需要优先完成以下判断：

- 当前投诉属于哪类问题
- 是否属于质量问题
- 是否在售后时效内
- 是否需要补材料
- 是否可继续处理

### 9.4 AI边界说明

第一版 AI 只做：

- 结构化理解
- 规则与 SOP 匹配
- 建议输出
- 动作推荐

第一版 AI 不做：

- 最终拍板
- 自动退款 / 自动退货
- 自动审批
- 自动执行闭环
- 自动回复

## 十一、规则逻辑

### 11.1 先补材料的情况

- 缺少订单或商品识别信息
- 客户问题描述过于模糊
- 缺少关键图片 / 视频 / 截图等证据
- 当前责任边界不清，无法判断归因

### 11.2 建议升级的情况

- 高金额或高风险投诉
- 规则冲突或无法明确归类
- 涉及安全或专业判断
- 多次投诉或历史争议单

### 11.3 可继续处理的情况

- 信息相对完整
- 证据基本齐全
- 规则明确
- 无需升级

### 11.4 继续处理路径

- 退款路径
- 退货退款路径
- 换货路径
- 补发路径

## 十二、状态流转说明

### 12.1 状态定义

第一版仅保留以下 4 个核心状态：

- 待处理：新投诉进入工作台，但尚未完成判断
- 待补材料：当前信息或证据不足，需要客户补充材料
- 待升级：当前问题复杂、风险较高，或规则冲突，需要升级处理
- 已标记处理：用户撤回申请、争议已解决，或人工确认当前工单关闭

### 12.2 状态流转

新投诉进入

↓

待处理

├─ 信息不足 → 待补材料

├─ 问题复杂 / 风险高 / 规则冲突 → 待升级

├─ 规则明确 → 进入继续处理路径

└─ 用户撤回 / 争议解决 → 已标记处理

### 12.3 继续处理路径说明

“继续处理”不是状态，而是从“待处理”状态进入的处理路径。

当 AI 判断当前信息相对完整、证据基本齐全且无需升级时，工单可进入以下推荐处理路径之一：

- 退款路径
- 退货退款路径
- 换货路径
- 补发路径

该路径用于指导一线售后执行下一步动作，不单独作为页面状态展示。

## 十三、MVP范围

### 13.1 本期范围

第一版仅验证“商品质量投诉分流”场景下的一线售后辅助闭环，当前原型覆盖以下模块与能力：

- 左侧工单展示侧边栏
- 中间原始信息区
- 历史记录与处理记录切换查看
- 右侧 Agent 辅助区
- 中间下半区客户沟通区
- mock 商品、订单、附件、聊天、处理记录数据
- AI 摘要、问题分类、SOP 判断依据、建议动作的前端展示
- 快捷操作推荐与基础状态切换

### 13.2 本期实现方式

- 使用 mock 数据完成页面展示与交互验证
- 使用 mock Agent 分析刷新模拟 AI 理解与建议输出
- 不接入真实企业系统、真实知识库或真实审批能力

### 13.3 非本期范围

- 登录注册
- 社区模块
- 完整售后工单系统
- 真实企业系统接入
- 真实知识库接入
- 自动审批
- 自动退款 / 自动退货
- 自动执行闭环
- 其他售后业务场景
- 权限后台
- 管理后台
- 报表系统
