# **02_业务理解**

1. ## **阶段目标**

本阶段的目标不是继续讲项目愿景，而是先搞清楚：

在没有 AI 的情况下，一线售后人员在“商品质量投诉分流”场景下到底是怎么处理问题的；他们依赖什么知识；哪一步最慢、最乱、最容易错；AI 最适合先插入哪一步。这个也符合手册对阶段二的要求：先理解当前售后处理流程、知识依赖、SOP 作用和高风险环节。 

1. ## **业务场景说明**

本项目总方向是 **企业售后****SOP****执行Agent**。

但为了控制第一版范围，我在当前阶段只聚焦一个切入场景：**商品质量投诉分流**。

这个场景下的核心用户仍然是一线售后处理人员。

他们面对的问题不是“完全不会处理”，而是当客户投诉内容模糊、证据不完整、规则边界复杂时，处理效率会明显下降，也更容易出现判断不一致、漏步骤和升级不及时等问题。这个判断和我在阶段一里确定的痛点一致：最难的不是机械执行，而是面对模糊投诉时，不知道该套哪条规则、该补哪些材料、是否需要升级。 

1. ## **当前业务流程（通用业务假设版）**

由于我没有真实售后岗位经验，以下流程不是某一家企业的真实内部 SOP，而是基于通用电商/企业售后逻辑建立的业务假设，用于定义第一版 AI 的介入点。

### **3.1 流程概述**

一线售后在处理质量投诉时，通常不是直接给出最终结论，而是先完成信息确认、证据核验、规则匹配和路径分流。结合当前项目范围，我将“无 AI 时”的流程先定义为以下 8 步：

### **3.2 当前流程**

#### **第一步：接收客户投诉**

客户通过聊天窗口、工单、平台留言或电话反馈商品问题。

客户表达通常不够标准，常见话术可能是“坏了”“有问题”“和图片不一样”“不能用”。

#### **第二步：初步理解投诉内容**

一线售后先快速判断客户大致在投诉什么，做一个初步分类。

这里一般不会直接下结论，只是先判断更接近哪类问题，例如：

- 明显破损 / 瑕疵
- 功能异常 / 无法使用
- 描述不符 / 边界模糊争议

#### **第三步：核对订单与商品信息**

售后确认当前投诉是否具备处理基础，通常会核对：

- 订单号
- 购买时间 / 收货时间
- 商品名称、规格、型号、颜色
- 是否仍在售后时效内
- 是否已有历史处理记录

如果订单信息或商品信息无法确认，流程不能继续深入，而是先进入待补充信息状态。

#### **第四步：核验证据材料是否齐全**

售后查看客户是否提供了足够证据支撑判断。

常见材料包括：

- 商品照片
- 破损细节图
- 功能异常视频
- 外包装图片
- 聊天截图
- 补充说明

如果证据不足，就不能直接进入处理结论，而应先补材料。

#### **第五步：对照规则与** **SOP**

当订单、商品和材料基本齐全后，一线售后开始结合规则判断：

- 是否属于质量问题
- 是否在处理时效内
- 是否满足退换、补发、维修等条件
- 是否还需要补充核验
- 是否需要升级处理

这一步最依赖经验，也最容易慢和乱。手册里也明确强调，阶段二要先搞清楚“根据什么规则处理”“哪些情况升级”。 

#### **第六步：做出初步分流判断**

一线售后不会立刻给出最终承诺，而是先把问题分流到某一条处理路径，例如：

- 信息不足，待补材料
- 规则明确，可继续处理
- 边界模糊，暂时无法明确判断
- 风险较高，建议升级处理

#### **第七步：回复客户并记录处理信息**

在路径明确后，售后需要同步完成两件事：

- 对客户说明下一步要求或处理意见
- 在内部记录投诉摘要、问题分类、证据情况和当前状态

#### **第八步：进入后续状态**

最后根据前面的判断进入不同状态，例如：

- 待补材料
- 处理中
- 建议升级
- 已完成
- 已关闭

这个状态结构也和手册后续场景建模里的状态示例一致。


1. ## **哪些情况必须先补材料**

这部分是流程里一个重要分支。结合你上一阶段的补充判断，我先整理为 4 类： 

### **7.1 缺少订单或商品识别信息**

没有订单号，或者无法确认商品具体型号、规格、款式时，一线售后无法进入实质判断。

### **7.2 客户问题描述过于模糊**

客户只说“坏了”“不行”“有问题”，但没有说明具体现象时，AI 或人工都很难准确分类。

### **7.3 缺少关键证据**

没有图片、视频、包装照片或故障表现证明时，质量问题很难核实。

### **7.4 责任边界不清**

当前信息无法判断是商品本身质量问题、运输问题，还是使用不当、人为损坏时，应先补材料，而不是直接给结论。

1. ## **哪些情况必须升级处理**

结合你在阶段一里已经定义过的升级条件，我在阶段二把它规范成更适合产品建模的版本： 

### **8.1 高金额或高风险投诉**

商品价值高、赔付风险高，或客户情绪激烈、反复投诉、可能引发更大影响时，不应由一线直接拍板。

### **8.2 规则冲突或无法明确归类**

现有信息下，投诉可能同时落入多个规则，或者无法判断属于质量问题、运输问题还是人为损坏时，需要升级。

### **8.3 涉及安全或专业判断**

如儿童使用安全、食品可食用性、电子产品使用风险等问题，超出普通一线售后处理边界时，应升级给更专业角色。

### **8.4 多次投诉或历史争议单**

客户已处理过一次但不满意，或同一订单存在历史争议记录时，也应进入升级或复核流程。

1. ## **当前业务依赖哪些知识**

手册要求阶段二建立知识分类意识，至少要区分规则类、流程类、话术类、案例类知识。 

结合你的项目，我把知识类型清单整理为下面 4 类：

| 知识类型   | 说明                     | 典型内容                                             |
| ---------- | ------------------------ | ---------------------------------------------------- |
| 规则类知识 | 决定能不能处理、如何处理 | 售后时效规则、质量判定规则、退换补发条件、升级标准   |
| 流程类知识 | 决定应该按什么顺序做     | 先核对什么、先补什么、什么情况下暂停、什么情况下升级 |
| 话术类知识 | 决定如何对客户表达       | 补材料提示话术、处理中回复话术、升级说明话术         |
| 案例类知识 | 帮助新手理解边界问题     | 类似投诉案例、典型误判案例、升级案例                 |

1. ## **SOP** **在这个场景里到底是什么**

在这个项目里，SOP 不是一篇很长的文档，而是一套结构化执行规则。手册对这一点说得很明确：SOP 至少要拆成触发条件、前置信息、判断规则、处理步骤、升级条件和输出结果。 

因此，在本项目第一版里，我将 SOP 理解为：

- 什么时候适用
- 需要先确认哪些信息
- 需要哪些材料
- 应该如何判断
- 下一步怎么走
- 什么情况下不能继续、要升级
- 最后给客户什么回复方向

1. ## **SOP** **结构模板**

为了便于后续做规则配置、AI 匹配和原型展示，我先统一定义每条 SOP 的结构模板如下：

| 字段         | 说明                       |
| ------------ | -------------------------- |
| SOP 名称     | 该 SOP 的名称              |
| 适用问题类型 | 对应哪类投诉               |
| 触发条件     | 满足什么条件后触发         |
| 必要输入信息 | 订单、商品、问题描述等     |
| 必要证据材料 | 图片、视频、截图等         |
| 判断规则     | 如何判断是否属于该 SOP     |
| 处理步骤     | 一线售后应按什么顺序处理   |
| 升级条件     | 哪些情况不能继续由一线处理 |
| 回复建议     | 对客户如何表达             |
| 风险提醒     | 容易误判或需要谨慎的地方   |

1. ## **示例** **SOP****（****Mock****）**

为了让后续原型和 workflow 更容易推进，这里先写 2 条 mock SOP 示例。

### **12.1 SOP-A：明显破损 / 瑕疵投诉初步分流**

**适用问题类型**

明显商品破损 / 瑕疵

**触发条件**

客户反馈商品存在肉眼可见破损、污渍、裂痕、零件脱落等问题

**必要输入信息**

订单号、商品名称、收货时间、问题描述

**必要证据材料**

商品整体图、破损细节图、外包装图

**判断规则**

若问题可通过照片初步验证，且在售后时效内，则进入质量投诉处理路径；

若无法判断破损来源，则先补材料或升级。

**处理步骤**

1. 核对订单与商品信息
2. 查看照片是否能明确识别问题
3. 判断是否处于售后时效内
4. 判断是否需要补充包装照片或更多细节图
5. 输出继续处理 / 补材料 / 升级建议

**升级条件**

高金额商品、责任边界不清、客户反复投诉

**回复建议**

先感谢反馈，再明确告知需要的材料或下一步处理路径

**风险提醒**

容易与运输破损混淆，不能只看客户一句话直接下结论

### **12.2 SOP-B：功能异常 / 无法使用投诉初步分流**

**适用问题类型**

商品功能异常 / 无法正常使用

**触发条件**

客户反馈商品无法开机、无法充电、按键失灵、无法正常运行等

**必要输入信息**

订单号、商品型号、使用时间、故障描述

**必要证据材料**

故障视频、操作过程说明、报错截图

**判断规则**

若客户仅口头描述故障但没有任何可验证证据，则先补充视频；

若问题涉及安全风险或专业判断，则升级。

**处理步骤**

1. 确认商品与订单信息
2. 要求客户描述故障现象
3. 索取视频或截图
4. 判断是否属于常见功能异常
5. 输出继续处理 / 补材料 / 升级建议

**升级条件**

涉及安全风险、规则冲突、无法判断原因

**回复建议**

引导客户补充更清晰的使用场景和故障材料

**风险提醒**

容易与客户使用不当混淆，一线不能直接做最终责任归因

1. ## **当前流程中最慢、最乱、最容易错的一步**

这一步必须由你自己先定，手册也特别强调这个判断不能丢给 AI。 

结合你在阶段一已经做出的判断，我在本阶段明确保留这个结论：

### **我认为最容易出错的一步是：**

**对照规则与** **SOP****，并判断是否需要补材料或升级。**

### **原因：**

- 客户原始描述通常不规范
- 信息和材料常常不完整
- 一线人员对规则理解深浅不同
- 同一问题可能存在多个解释路径
- 一旦误判，后面回复、升级和记录都会跟着错

这也解释了为什么第一版 AI 最适合先做“结构化理解 + 规则匹配 + 建议输出”，而不是直接替代人工拍板。手册对第一版边界也强调过：AI 只做建议，不做最终拍板。 

1. ## **AI 在阶段二识别出的最适合介入点**

基于以上业务理解，我认为 AI 最适合优先介入的是以下 4 个点：

1. **投诉摘要**

把客户原始表述整理成更容易处理的结构化信息

1. **问题分类**

先判断更接近哪一类质量投诉

1. **补材料提示**

自动识别当前材料缺口，提示一线向客户追问

1. **推荐** **SOP** **/ 下一步动作**

给出“继续处理 / 补材料 / 升级”的路径建议

这也和你阶段一已经确定的四个核心输出模块一致。 

1. ## **本阶段结论**

通过阶段二的业务理解，我目前形成的判断是：

- 第一版不需要假装覆盖完整售后执行流程
- 只聚焦“商品质量投诉分流”已经足够支撑一个完整的 AI 产品切口
- 当前最值得优先解决的，不是最终处理结论，而是模糊投诉下的信息理解、材料补齐、规则匹配和路径分流
- 第一版 AI 应该先作为一线售后的辅助分诊层，而不是自动拍板层
